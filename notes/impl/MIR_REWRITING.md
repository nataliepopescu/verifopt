# mir rewriting

MIR rewriting happens in two passes because of some odd constraints with how `rust_driver::Callbacks` works. the first pass is the normal one which checks for potential rewrite places using FSA, which now gives an output of a function (keyed by `DefPathHash` to survive the two pass approach) and the block which has the jump which can be made into static calls. then, moving onto the actual rewrite part, these edits are separated into two edit types: a single `Single(DefPathHash)` substitution or multiple ones with `Multi(Vec<DefPathHash>)`. for the former, the rewrite simply replaces the dynamic function call with the correct single one based on `DefPathHash`.

for the latter, multiple if terminators are used for calling the correct function, using the vtable function pointer of the incoming value for the discriminator. the edit itself starts with a bunch of casts to get a `*const (fn ptr)`, which is then offset by the vtable slot for the certain call that is happening. then, blocks are created in reverse, starting with a final block that uses one version of the call, and then a block before that with an if to either a different call or that final block, and so one. however, the start is also a special case, as the if terminator is inserted directly in place in the original block that is being edited. finally, because of how calls work in MIR, each static call uses the same `target` based on the original dynamic call which makes it trivial to continue logic past the rewrite.
